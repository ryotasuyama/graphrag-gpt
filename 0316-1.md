# 0316-1 修正計画書

## 1. 目的

`query.py` のブラケット自動修正フローにおいて、同系統の `CreateBracket` エラーを繰り返す問題を減らす。
特に以下の3点を同時に強化する。

- プロンプトで修正範囲を広げる
- 失敗候補の構造化情報を LLM に渡す
- 同系統の失敗候補を再生成しないよう禁止条件を入れる


## 2. 背景

現状の修正フローは、主に「エラー行周辺の `bracketPramN` を見つけ、その候補だけを修正する」前提で設計されている。
このため、以下のような問題が起きている。

- `pywintypes.com_error` のような一次障害には対応しやすい
- `NameError: bracketX is not defined` のような二次障害は、実際の失敗原因まで遡れず取りこぼしやすい
- 失敗候補と近縁の候補を LLM が再生成し、同じ種類の COM 例外を繰り返す
- Fore/Aft 整合だけでなく、面種別、部材型、成立実績まで見ないと直らないケースがある

したがって、「修正能力が足りない」というより、「修正対象の定義と再試行制約が狭い」ことが主因である。


## 3. 現状課題

### 3.1 修正対象のスコープが狭い

- 失敗した1行のみを見て修正しようとする
- 失敗候補の近縁パターンまで除外・置換できていない
- ループ単位の危険な生成方針を止められていない

### 3.2 失敗情報が LLM に十分伝わっていない

- `stderr` は渡しているが、失敗候補の署名が明示されていない
- `Surfaces1` / `Surfaces2` / `BaseElement` / `BracketType` の組が禁止条件として整理されていない

### 3.3 同系統の再発防止が弱い

- 一度失敗した候補を、別名や別候補番号で再生成する余地がある
- `Deck.D` の成功パターンを `Deck.A/B/C` に機械的に横展開してしまう
- Side 系で未実績の `1501 / FL-FL` を再採用しやすい

### 3.4 実行前チェックが不足している

- `bracketX` 未定義のまま `SetElementColor` などを呼ぶ構造を事前検知できていない
- `1505` に必要な `Sf1EndElements` の欠落を実行前に弾けていない


## 4. 改修方針

改修は以下の3本柱で行う。

### 方針A: 修正対象の範囲を広げる

- 失敗した1候補だけでなく、その近縁候補の除外・置換も許可する
- `NameError` を二次障害として再分類し、対応する `CreateBracket` 候補の修正へ変換する
- 必要に応じて後段の色設定や表示制御も修正対象に含める

### 方針B: 失敗候補を構造化して LLM に渡す

- 単なる traceback ではなく、失敗候補の署名を抽出して渡す
- 署名は再生成禁止ルールにも使える形式で保持する

### 方針C: 同系統の再生成を禁止する

- 過去に失敗した候補を blacklist 化する
- 同一または近縁の候補を再生成しないよう、再分析・再生成プロンプトへ明示する
- 実行前に blacklist と静的チェックを通す


## 5. 改修項目

### 5.1 エラー分類の見直し

#### 目的

二次障害を一次障害へ正規化し、実際に直すべき `CreateBracket` 候補へ到達できるようにする。

#### 対応内容

- `pywintypes.com_error`
  - 従来通り `CreateBracket` 失敗として扱う
- `NameError: bracketX is not defined`
  - 二次障害として扱う
  - `bracketX` をキーに、対応する `bracketX = part.CreateBracket(...)` を逆引きする
  - 対応する `bracketPramN` を特定し、一次障害として再構成する
- 逆引きできない場合
  - 生成漏れとして扱う
  - `bracketX` の使用箇所と生成箇所の不整合を LLM へ明示する

#### 期待効果

- `NameError` をそのまま修正するのではなく、本当の失敗候補に対して再試行できる


### 5.2 失敗候補情報の構造化

#### 目的

LLM が「何が失敗したか」を具体的に判断できるようにする。

#### 追加する情報

- `failed_bracket_var`
- `failed_bracket_param`
- `failed_signature`
- `BracketType`
- `BaseElement`
- `Surfaces1`
- `Surfaces2`
- `Sf1EndElements`
- 関連要素名
- 関連 `ProfileType`
- 失敗候補の前後コンテキスト

#### 署名フォーマット案

```text
1501|Base=profile77[0]|S1=profile77[0],FL|S2=profile41[0],FL
```

#### 期待効果

- 失敗候補を曖昧な文章ではなく、比較可能な構造として扱える
- blacklist や近縁判定に流用できる


### 5.3 再分析プロンプトの拡張

#### 目的

LLM に「どこまで直してよいか」を明確に伝える。

#### 追加指示

- 失敗候補そのものを除外してよい
- 同系統の未実績候補も除外してよい
- `1501 / FL-FL` が不成立なら `1505 / PLS-FL + Sf1EndElements` へ置換してよい
- 候補数要件は危険候補の追加ではなく、成功実績候補の維持で満たしてよい
- `NameError` の場合は未定義変数使用行ではなく、対応するブラケット生成候補を修正対象とする
- 成功済み候補は原則維持し、失敗候補とその近縁の危険候補のみ修正する

#### 期待効果

- LLM が「失敗候補を残したまま別の微修正だけ行う」ことを減らせる


### 5.4 再生成禁止ルールの導入

#### 目的

同じ失敗を別名で繰り返すことを防ぐ。

#### 対応内容

- `error_history` に `failed_signature` を蓄積する
- 再分析・再生成プロンプトに以下を明記する
  - 同一署名の候補を再生成しない
  - 近縁候補も再生成しない

#### 近縁候補の判定ルール案

- 同じ `BaseElement`
- 同じ `Surfaces2` 相手
- 同じ `BracketType`
- 同じ `FR` または `DL` 系列
- 同じ `ProfileType` 組み合わせ

#### 期待効果

- 同じ COM 例外候補の再発率を下げられる


### 5.5 実行前 validator の追加

#### 目的

実行してから分かる単純ミスを事前に止める。

#### チェック内容

- `SetElementColor(bracketX, ...)` の前に `bracketX = part.CreateBracket(...)` が存在するか
- `BlankElement(bracketX, ...)` の対象が定義済みか
- `1505` 候補に `Sf1EndElements` があるか
- blacklist 済み署名が含まれていないか
- 明らかな禁止パターンが含まれていないか

#### 禁止パターン例

- Side 系の未実績 `1501 / FL-FL`
- `Deck.D` 成功型の `Deck.A/B/C` への機械的横展開
- `Fore` に対する `A/AP`、`Aft` に対する `F/FP`

#### 期待効果

- `NameError` や単純な構成不備を実行前に減らせる


### 5.6 ドメイン制約の明文化

#### 目的

既存の成功実績ベースのルールを、再試行時にも崩さないようにする。

#### 明文化するルール

- `Wall.Fore.DLxx.*` は `Deck.*.DLxx.F / FP` のみ
- `Wall.Aft.DLxx.*` は `Deck.*.DLxx.A / AP` のみ
- `Deck.D` の成功実績を `Deck.A/B/C` に自動横展開しない
- `1002 / 1003 / 1201` の部材型差を必ず考慮する
- Side 系では未実績 `1501` より `1505` を優先する
- `WF / WB` は成功実績のあるケースに限定する
- `1505` では `Sf1EndElements` を省略しない

#### 期待効果

- 再分析で危険な候補が復活しにくくなる


## 6. 実装順

### フェーズ1: 最低限の再発防止

1. `NameError` の再分類処理を追加する
2. `failed_signature` を抽出して `error_context` に入れる
3. 再分析・再生成プロンプトに禁止条件を追加する

### フェーズ2: 再試行品質の安定化

4. blacklist ベースの再生成禁止を導入する
5. 実行前 validator を追加する

### フェーズ3: 精度向上

6. 近縁候補判定の精度を上げる
7. 部材型や系列単位での危険パターン判定を強化する


## 7. 優先度

### 高

- `NameError` 再分類
- 失敗候補署名の構造化
- プロンプトへの再生成禁止条件追加

### 中

- blacklist 導入
- 実行前 validator

### 低

- 近縁候補の高度な自動判定
- ドメイン制約の自動抽出強化


## 8. 成功条件

以下を満たせば、本改修は有効と判断する。

- `NameError: bracketX is not defined` が、未定義変数行ではなく対応するブラケット生成候補の修正につながる
- 同一署名または近縁署名の候補を再試行で再生成しない
- `1505` の `Sf1EndElements` 欠落や、定義前の `bracketX` 使用を実行前に検知できる
- 同じ COM 例外の繰り返し回数が減る
- 成功済み候補を保ったまま、失敗候補のみを除外・置換できる


## 9. 想定リスク

- blacklist を厳しくしすぎると、必要候補まで除外する可能性がある
- 近縁判定が粗いと、本来成立する候補も巻き込む可能性がある
- プロンプト拡張だけでは、完全に同系統再発を防げない可能性がある
- validator の判定が弱いと、依然として COM 例外は残る

対策として、初期段階では blacklist と validator を保守的に導入し、失敗履歴から徐々に判定を強化する。


## 10. 補足

今回の中心課題は「LLM の修正能力不足」ではなく、以下の3点である。

- 修正対象の定義が狭い
- 失敗情報が構造化されていない
- 再発禁止ルールが弱い

したがって、本計画ではプロンプト改善のみではなく、失敗候補情報の構造化と実行前チェックを含めて改修する。
