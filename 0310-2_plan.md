# 実装計画: query.py の検索カバレッジ拡張

作成日: 2026-03-10

---

## 背景

`query.py` は EvoShip CADスクリプト生成用のシステムであり、
`ask()` → `_retrieve_help_context()` → Neo4j クエリ → LLM プロンプトへ付加、という流れでヘルプ情報を活用している。

調査の結果、以下のギャップが判明した。

| ギャップ | 現状 | 影響 |
|---------|------|------|
| `DimensionType` が未クエリ | スキーマ定義（テンプレート文字列）には記載あり、実クエリなし | ブラケットタスク時に Sf1/Sf2 の寸法型情報がプロンプトに入らない |
| `DESCRIBES` リレーションが未活用 | スキーマ定義には記載あり、実クエリなし | キーワード検索でヒットした HelpPage がどの Method/Object を説明しているか LLM に伝わらない |

---

## 修正対象ファイル

### `query.py` の `_retrieve_help_context()` のみ修正する

---

## 修正内容（詳細）

### 1. ブラケットタスク時に `DimensionType` を追加取得

**対象箇所**: `_retrieve_help_context()` 内の `if is_bracket:` ブロック末尾。

現在 `BracketShapeType` を取得した直後に、`DimensionType` の全件取得クエリを追加する。

```cypher
MATCH (d:DimensionType)
RETURN d.dim_type   AS dim_type,
       d.shape_name AS shape_name,
       d.params     AS params,
       d.usage_note AS usage_note
ORDER BY d.dim_type
```

取得後、以下のような Markdown セクションを `sections` リストに追加する:

```
### Sf1/Sf2 DimensionType 一覧
- Type 1531 (A) params=[D, L]
- Type 1541 (E2) params=[G, L]
...
```

`DimensionType` が 0件（未挿入）の場合はそのセクションをスキップする（既存の BracketShapeType 処理と同じパターン）。

---

### 2. キーワード検索（共通）に `DESCRIBES` 順方向を追加

**対象箇所**: `_retrieve_help_context()` 内のキーワード検索ループ。

現在のクエリ（HelpPage のタイトル/summary を返すのみ）を以下のように拡張する。

```cypher
-- 修正後
MATCH (h:HelpPage)
WHERE any(kw IN h.keywords WHERE toLower(kw) CONTAINS toLower($term))
   OR toLower(h.title) CONTAINS toLower($term)
OPTIONAL MATCH (h)-[:DESCRIBES]->(target)
RETURN h.title AS title, h.summary AS summary,
       collect(target.name) AS describes_targets
LIMIT 5
```

`sections` に追加するフォーマットを更新し、`describes_targets` が存在する場合は以下のように表示する:

```
### 関連ヘルプページ（キーワード: bracket）
- [ブラケット] ブラケットを作成します。 → 関連API: CreateBracket, ShipBracket
```

`describes_targets` が空リストの場合は "→ 関連API: ..." の部分を省略する。

## 修正しない箇所

| 箇所 | 理由 |
|------|------|
| `CYPHER_GENERATION_TEMPLATE_JP` のスキーマ記述 | DimensionType / DESCRIBES はすでに記載済み |
| `ask()` | help_ctx の付加ロジックは変更不要 |
| `_extract_search_terms()` | 変更不要 |
| BracketShapeType 取得ロジック | 既に正しく実装されている |
| `is_bracket_task()` 判定 | `prompts/loader.py` 側で定義済み、変更不要 |

## 修正後の動作イメージ

ブラケット追加指示が来た場合、`ask()` に渡る help_ctx が以下のように拡張される:

```
## ヘルプDB参照コンテキスト（Neo4jから取得）

### ブラケット形状種別（BracketShapeType）
- Type 1501 (2-B) [2面] params=[なし] ※Profile FL × Profile FL の標準形
- Type 1505 (2-E) [2面] params=[R]    ※Plate PLS × Profile FL の標準形
...

### Sf1/Sf2 DimensionType 一覧           ← 今回追加
- Type 1531 (A) params=[D, L]
- Type 1541 (E2) params=[G, L]
...

### ブラケットコマンド説明
ブラケットを作成します。（２面、３面間に板形状を作成します。）

### 関連ヘルプページ（キーワード: bracket）
- [ブラケット] ブラケットを作成します。 → 関連API: CreateBracket  ← 今回追加
```

## 作業順序

1. `_retrieve_help_context()` の `if is_bracket:` ブロックに DimensionType クエリを追加
2. キーワード検索ループのクエリに `OPTIONAL MATCH (h)-[:DESCRIBES]->(target)` を追加
3. `sections.append(...)` のフォーマット文字列に `describes_targets` を組み込む
4. 動作確認:
   ```bash
   # query.py を直接呼び出すテスト（またはスクリプト生成を通じて確認）
   python -c "
   from query_graph import _retrieve_help_context
   ctx = _retrieve_help_context('ブラケットを追加してください')
   print(ctx)
   "
   ```
   出力に `DimensionType 一覧` セクションが含まれることを確認。

## `help_qa.py` との対応関係

同じギャップを `help_qa.py`（0310-1_plan.md）でも修正する。
両ファイルで対称的に実装することで、スクリプト生成用途・QA用途の両方でカバレッジが揃う。

| 機能 | help_qa.py | query.py |
|------|-----------|----------------|
| BracketShapeType 取得 | search_bracket_types() で新規追加 | 既に実装済み |
| DimensionType 取得 | search_bracket_types() 内で追加 | _retrieve_help_context() に追加 |
| DESCRIBES 順方向 | search_help_pages() クエリ拡張 | キーワード検索クエリ拡張 |
| DESCRIBES 逆方向 | search_api_methods() クエリ拡張 | （スコープ外：スクリプト生成では不要） |
