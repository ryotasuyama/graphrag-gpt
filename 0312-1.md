# 0312-1 修正計画

## 目的

`help/` 配下のHTMLヘルプを、そのまま検索用テキストとして扱うのではなく、`query.py` のコード生成に直接効く「構造化されたCAD知識」として使えるようにする。

特にブラケット生成に関して、以下を改善対象とする。

- `help` の説明文を、コード生成ルールとして再利用できる形に変換する
- `query.py` の既定経路でも、その知識が確実に使われるようにする
- `summary/raw_text` 中心の曖昧な参照ではなく、型番・必須フィールド・禁止条件・選択条件を明示的に扱う

---

## 現状整理

### 1. ingest 経路が分かれている

- `ingest.py`
  - API仕様 (`data/api1.txt` など) と `api_arg.txt` をNeo4j/Chromaに投入している
  - `help/` は取り込まない
- `ingest_html.py`
  - `help/` を解析して `HelpPage` を作る
  - `prompts/bracket_type_reference.md` から `BracketShapeType` / `DimensionType` を作る
  - ただし抽出の主眼は `summary` / `keywords` / `raw_text` であり、コード生成ルールとしては弱い

今回の修正では、`ingest_html.py` を主軸の取り込み経路として扱う。
`ingest.py` 側への統合はこの計画の対象外とし、必要なら後続で README / 運用手順を更新して明示する。

### 2. `query.py` は help を部分的にしか使っていない

- `ask()` 経由では `_retrieve_help_context()` により help をプロンプトへ追加している
- しかし既定の `pipeline-mode=three` では、初回生成は主に以下の流れで進む
  - `analyze_bracket_placement()`
  - `generate_bracket_section()`
- この経路では、help 由来の構造化知識が明示的には渡っていない

### 3. いまの `HelpPage` は「説明用」には有効だが「生成用」には弱い

有効なケース:

- `help_qa.py` のようなヘルプ検索・説明回答
- コマンド説明やUI概念の参照

弱いケース:

- `CreateBracketParam` の必須フィールド決定
- `BracketType` の選択
- `Sf1/Sf2 DimensionType` のデフォルト決定
- 3面ブラケット未対応の回避
- 禁止パターンの明示

---

## 方針

### 基本方針

`help/` から抽出した知識を、以下の2層に分ける。

1. 検索・説明向け知識
   - 従来どおり `HelpPage`
   - `title`, `summary`, `keywords`, `raw_text`

2. コード生成向け知識
   - 正本が明示された情報だけを、ルール化されたノード/プロパティとして保存
   - `query.py` の分析・生成プロンプトに直接渡す

つまり、「ヘルプページ全文を読ませる」のではなく、「コード生成に必要な事実だけをルールとして渡す」形に変える。

### 正本の扱い

Neo4j に挿入してよいのは、以下のいずれかに該当する情報だけとする。

1. `prompts/bracket_type_reference.md` に明示されており、正規表現などで決定論的に抽出できるもの
2. `help/cmd_ship_bracket.html` に明示されており、HTML解析で決定論的に抽出できるもの
3. 既存プロンプト群のうち、将来「構造化知識の正本」として採用する専用ファイルに定義されているもの

逆に、以下はこの計画では Neo4j に入れない。

- LLM が推測しただけの選択ルール
- 参考コードから統計的に推定したデフォルト値
- 正本ファイルが未定義の禁止事項

## 追加・変更したい知識の形

### A. ブラケット型番ルール

対象:

- `BracketShapeType`
- `DimensionType`

この層は既に `prompts/bracket_type_reference.md` があるので、まずはこの情報を起点に強化する。

保存したい例:

```json
{
  "id": "BracketShapeType_1505",
  "type": "BracketShapeType",
  "properties": {
    "bracket_type": 1505,
    "shape_name": "2-E",
    "category": "2面",
    "surface_pair_hint": ["PLS", "FL"],
    "required_fields": [
      "BaseElement",
      "Surfaces1",
      "Surfaces2",
      "Thickness",
      "BracketParams",
      "Sf1DimensionType",
      "Sf2DimensionType"
    ],
    "default_bracket_params": ["200"],
    "default_sf1_dimension_type": 1541,
    "default_sf1_dimension_params": ["0", "100"],
    "default_sf2_dimension_type": 1531,
    "default_sf2_dimension_params": ["200", "15"],
    "usage_note": "Plate PLS × Profile FL の標準形"
  }
}
```

### B. コマンド由来の生成ルール

対象:

- `help/cmd_ship_bracket.html`

抽出したい内容:

- 2面ブラケットでは `Surfaces1`, `Surfaces2` が必要
- 3面ブラケットでは `Surfaces3` が必要
- 寸法基準1/2が必要
- 形状により追加基準面が必要な場合がある
- フランジの有無
- 端部スカラップの有無

保存例:

```json
{
  "id": "Rule_CreateBracket_SurfaceRequirements",
  "type": "GenerationRule",
  "properties": {
    "target_api": "CreateBracket",
    "rule_kind": "required_fields",
    "when_category": "2面",
    "required_fields": ["Surfaces1", "Surfaces2"]
  }
}
```

```json
{
  "id": "Rule_CreateBracket_Surface3Required",
  "type": "GenerationRule",
  "properties": {
    "target_api": "CreateBracket",
    "rule_kind": "required_fields",
    "when_category": "3面",
    "required_fields": ["Surfaces1", "Surfaces2", "Surfaces3"]
  }
}
```

### C. 選択ルール

ブラケット生成で特に重要な条件分岐を、自然文ではなく条件ルールとして保存する。

ただし、`SelectionRule` として保存するのは、正本ファイルで明示されている条件だけに限定する。
たとえば `PLS × FL -> 1505` や `FL × FL -> 1501` は、`prompts/bracket_type_reference.md` や既存ブラケット規則ファイルに明示されている場合のみ保存対象とする。

保存例:

```json
{
  "id": "Rule_BracketType_PLS_FL",
  "type": "SelectionRule",
  "properties": {
    "target_api": "CreateBracket",
    "condition_surface_pair": ["PLS", "FL"],
    "preferred_bracket_type": 1505
  }
}
```

```json
{
  "id": "Rule_BracketType_FL_FL",
  "type": "SelectionRule",
  "properties": {
    "target_api": "CreateBracket",
    "condition_surface_pair": ["FL", "FL"],
    "preferred_bracket_type": 1501
  }
}
```

### D. 禁止ルール

既存プロンプトにある禁止事項を、Neo4jにも持たせる。

ただし、禁止ルールも正本が未定義なら投入しない。
現時点では、専用の正本ファイルを定めずに `prompts/bracket_domain_rules.md` の文面をそのまま複製することはしない。
将来的に禁止事項を Neo4j に入れる場合は、決定論的に抽出できる専用ソースを先に定義する。

保存例:

```json
{
  "id": "Rule_Bracket_CreateFlag",
  "type": "ForbiddenRule",
  "properties": {
    "target_api": "CreateBracket",
    "forbidden_pattern": "part.CreateBracket(bracketParam, True)",
    "reason": "ブラケットが非表示で作成される"
  }
}
```

```json
{
  "id": "Rule_Bracket_BlankElement",
  "type": "ForbiddenRule",
  "properties": {
    "target_api": "BlankElement",
    "forbidden_pattern": "part.BlankElement(bracketX, True)",
    "reason": "ブラケットが見えなくなる"
  }
}
```

## 編集対象ファイル

### 1. `ingest_html.py`

最重要。`help` から「構造化ルール」を抽出する中心。

変更内容:

- `HelpPage` の保存は維持する
- 新しいノード型を追加する
  - `GenerationRule`
  - `SelectionRule`
  - `ForbiddenRule`
- `cmd_ship_bracket.html` など特定の重要ページから、ルールを決定論的に抽出する処理を追加する
- `BracketShapeType` / `DimensionType` に、生成向けプロパティを追加する
  - `surface_pair_hint`
  - `required_fields`
  - `default_*`
  - `supports_flange`
  - `supports_end_scallop`
- 既存の `HelpPage -> DESCRIBES` だけでなく、
  - `HelpPage -> DEFINES -> BracketShapeType`
  - `HelpPage -> DEFINES -> GenerationRule`
  - `BracketShapeType -> HAS_RULE -> SelectionRule`
  のような関係を追加することを検討する

実装ポイント:

- まずはLLM抽出ではなく、`cmd_ship_bracket.html` と `bracket_type_reference.md` を対象に、正規表現・HTML解析ベースで決定論的に組む
- 曖昧な文書全般にLLMをかけるのは後回し
- 最初の対象はブラケット関連のみでよい
- `ingest_html.py` を主軸の ingest とし、この計画では `ingest.py` への統合は行わない
- 正本が未定義のルールはノード化しない

### 2. `query.py`

構造化知識を、実際のコード生成経路で使うために必要。

変更内容:

- `_retrieve_help_context()` を拡張し、`HelpPage` だけでなく新しいルールノードも取得する
- 取得対象の優先順を分ける
  - 候補面ペアや `reference_code` から関連が高い `BracketShapeType`
  - その `BracketType` に対応する `DimensionType`
  - 正本が明示された `SelectionRule` / `GenerationRule`
  - 補助的に `HelpPage`
- `ask()` ではなく、既定の `pipeline-mode=three` の**初回生成経路のみ**でこの知識を使う
  - `analyze_bracket_placement()` に渡す指示文へ構造化知識を追加する
  - `generate_bracket_section()` にも必要最小限の要約を渡す
- 自己修正ループ（`reanalyze_bracket_placement()` / `generate_bracket_section_with_error()`）への適用は今回の計画対象外とする

実装ポイント:

- 既定経路の `analyze_bracket_placement()` が help なしで走っている点を解消する
- 「大量のhelp本文」ではなく「ルール一覧の短い整形テキスト」を渡す
- 全件取得はしない。候補面ペアや `reference_code` から `1501` / `1505` 周辺だけに絞る
- `DimensionType` も、その `BracketType` に実際に関連するものだけに絞る
- 取得件数に上限を設け、不要な `raw_text` は渡さない

### 3. `prompts/loader.py`

必要なら、構造化知識をプロンプトへ差し込むためのヘルパー関数を追加する。

変更候補:

- `build_analysis_prompt()` に `knowledge_section` を追加
- `build_bracket_section_prompt()` に `knowledge_section` を追加
- あるいは専用の整形関数を追加して、`query.py` 側で文字列化したものを渡す

注意:

- 今回の対象は `pipeline-mode=three` の初回生成に使う `build_analysis_prompt()` / `build_bracket_section_prompt()` のみ
- リトライ用プロンプトビルダはこの計画では変更対象に含めない

### 4. `query_graph.py`

`query.py` に合わせて整合させる必要がある。

変更内容:

- `query.py` で導入したルールノードをこちらでも認識できるようにする
- Cypher生成プロンプトのスキーマ説明に新ノード型を追加する

注意:

- `query_graph.py` は `HelpPage` / `BracketShapeType` / `DimensionType` を明示している
- 今後新ノードを導入するなら、このファイルも追従が必要

### 5. `help_qa.py`

必須ではないが、help検索との役割分担を整理するために確認対象。

変更方針:

- 基本は説明用途のまま
- 必要なら `GenerationRule` 等も参照できるようにする
- ただし優先度は低い

### 6. `ingest.py`

今回の計画では変更対象外。

方針:

- ブラケット関連の構造化知識取り込みは `ingest_html.py` を主軸とする
- `ingest.py` への統合や経路一本化は後続タスクとして扱う
- 必要なら README / 運用手順の明記だけを後で行う

## 具体的な修正イメージ

### 修正イメージ 1: `HelpPage` 中心から `Rule` 中心へ

現状:

- `HelpPage.summary` を検索して、プロンプトへ追記する

変更後:

- 正本が定義された `SelectionRule` / `GenerationRule` を優先して取得する
- `HelpPage.summary` は補足説明としてだけ使う
- 正本未定義の `ForbiddenRule` はこの段階では取得対象に含めない

### 修正イメージ 2: ブラケット型番ノードの強化

現状:

- `BracketShapeType`
  - `bracket_type`
  - `shape_name`
  - `bracket_params`
  - `category`
  - `usage_note`

変更後:

- 上記に加えて
  - `surface_pair_hint`
  - `required_fields`
  - `default_bracket_params`
  - `default_sf1_dimension_type`
  - `default_sf1_dimension_params`
  - `default_sf2_dimension_type`
  - `default_sf2_dimension_params`
  - `supports_flange`
  - `supports_end_scallop`

### 修正イメージ 3: `query.py` の既定経路で知識を使う

現状:

- `ask()` だけは help を使う
- `pipeline-mode=three` の初回分析は使わない

変更後:

- `process_generation_loop()` の初回生成前に、ブラケット用の構造化知識を取得する
- それを `analyze_bracket_placement()` と `generate_bracket_section()` のプロンプトへ差し込む
- 対象は候補面ペアや `reference_code` から関連が高い `1501` / `1505` 周辺に限定する

## 検証項目

### 1. Neo4j投入結果の確認

確認内容:

- `BracketShapeType` に追加プロパティが入っているか
- 正本が明示された `GenerationRule`, `SelectionRule` が作成されているか
- `ForbiddenRule` は、正本が定義された場合のみ作成されているか
- `cmd_ship_bracket` と各ルールが正しく接続されているか

### 2. `query.py` の既定経路で利用されるか

確認内容:

- `pipeline-mode=three` で初回生成時に構造化知識がAgent 1へ渡るか
- Agent 2にも必要最小限の知識が渡るか
- 不要な `BracketShapeType` / `DimensionType` 全件が渡っていないか
- 候補面ペアや `reference_code` に応じて `1501` / `1505` 周辺だけに絞られているか

### 3. ブラケット生成品質の確認

確認観点:

- `PLS × FL` のとき `1505` を優先できるか
- `FL × FL` のとき `1501` を優先できるか
- `Sf1/Sf2 DimensionType` の既定値が安定するか
- 3面ブラケット未対応ケースを避けられるか
- `CreateBracket(..., False)` が守られるか

### 4. 過剰投入の抑制

確認内容:

- `raw_text` の大量注入でプロンプトが肥大化していないか
- 説明文よりルールが優先されているか
- 取得件数上限が効いているか
- 1501/1505 と無関係な型番が混入していないか

## 実施時の注意

- 最初から全 help を高度に構造化しようとしない
- まずはブラケット関連に限定する
- Neo4j に入れるのは、正本が明示され決定論的に抽出できる事実だけに限定する
- 取得コンテキストは候補面ペアや `reference_code` を用いて `1501` / `1505` 周辺へ絞り、全件列挙を避ける
- 今回のプロンプト接続対象は `pipeline-mode=three` の初回経路のみとする
- `LLMで要約` より `決定論的な抽出` を優先する
- 生成用知識と説明用知識を混ぜない
- `query.py` の既定経路に接続しない限り、取り込みだけ改善しても効果は限定的

## 最終的な到達点

理想形は以下。

- `help/` は「ヘルプ全文検索DB」ではなく、「CAD生成ルールのソース」になる
- `query.py` は `HelpPage.summary` ではなく、抽出済みルールを主に参照する
- ブラケット生成で必要な判断
  - 型番選択
  - 必須フィールド補完
  - デフォルト値選択
  - 禁止条件回避
  が、自然文の解釈任せではなく、構造化知識に基づいて行われる

この状態になれば、`help` をグラフへ入れる意味が、単なる説明補助ではなく、コード生成品質の改善として明確に出る。
