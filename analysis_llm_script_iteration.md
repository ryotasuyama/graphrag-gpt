# 分析：LLMスクリプト生成の繰り返しは解を収束させるか？

## 観察されたエラーパターン

| バージョン | 失敗ブラケット | エラー |
|-----------|--------------|--------|
| 0226-3 | #1（最初）| COM -2147467259 |
| 0226-4 | #18（途中）| COM -2147467259 |

同一の COM エラーが繰り返し発生している。エラーメッセージに有用な情報はない。

---

## 中心的な問い：なぜ繰り返しが機能しないか

### フィードバックの質が根本問題

LLMが受け取る情報：
- `pywintypes.com_error: (-2147467259, '例外が発生しました。', ...)` のみ
- どのパラメータが間違っているか？ → 不明
- 正しい値は何か？ → 不明
- 他の何かを変えれば良いのか？ → 不明

これは「正解/不正解」しか返さないテストに対してランダムに回答を変え続けるようなもの。
**方向性のない探索（Random Walk）**であり、収束しない。

---

## 知識ギャップの分類

### A. 反復で補える知識（Recoverable）

| 不足知識 | 理由 |
|---------|------|
| FlangeParams の典型値 | 参考スクリプトのパターンが学習できる |
| BracketType の使い分け (1501/1505) | analysis.md に書かれており学習済み |
| Scallop1Type / Sf1DimensionType | 参考スクリプトで確認可能 |

これらは「参考スクリプトに存在するパターン」であり、エラーが起きなければ検証可能。

### B. 反復では補えない知識（Non-Recoverable）

| 不足知識 | 理由 |
|---------|------|
| PLS形式のベクトル値 (nx, ny, nz) | 実際の3Dジオメトリに依存。モデルを見ないと算出不可 |
| Sf1EndElements の設定値 | CADモデルのトポロジー（どのエレメントが隣接しているか）が必要 |
| BracketParams=["200"] の必須条件 | EvoShip内部仕様。ドキュメント化されていなければ不明 |
| profile[0] vs profile[1] の選択基準 | API仕様書の深い理解が必要。参考スクリプトだけでは判断不能 |

---

## 反復回数の効果についての推論

### 4回以上繰り返せば完成するか？

**短い答え：おそらくNo**

**理由：**

1. **探索空間の爆発**
   - 各ブラケットに約10個のパラメータ、各パラメータに数十の候補値
   - 20ブラケット × 10パラメータ = 200変数を同時最適化
   - LLMは複数変数を同時に変更するため、何が効いたか特定できない

2. **0226-3→0226-4の「改善」が実は問題**
   - #1で失敗→#18で失敗という変化は「改善」に見えるが、
   - 実際は #1-17 がたまたま正しかっただけかもしれない
   - 次のバージョンで #1 から失敗が再発する可能性がある

3. **ゼロ情報フィードバック**
   - COM エラーは「失敗した」という事実だけを伝える
   - LLMには「何を変えれば良いか」の手がかりが全くない
   - 人間の試行錯誤でも、このフィードバックではデバッグは困難

---

## 根本原因：アーキテクチャ設計の問題

現在の構造：
```
LLM → スクリプト生成 → 実行 → COM error → LLM（同じ情報で再生成）
```

この構造には2つの欠陥がある：

**欠陥1：ジオメトリ情報の欠如**
LLMはEvoShipのCADモデルに実際にアクセスできない。
PLS法線ベクトルは「プログラムで計算できるもの」ではなく「CADモデルを見て決めるもの」。

**欠陥2：エラー診断の欠如**
失敗したCreateBracket呼び出しのどのパラメータが問題か、
EvoShip側からフィードバックされない。

---

## 解決アイデア（根拠付き）

### アイデア1：バイナリサーチ + 個別テスト法
**内容：** 一度に全ブラケットを生成せず、1つずつ生成・テストするスクリプトを書く
**根拠：** どのブラケットがどのパラメータで失敗するかを特定できる
**限界：** フィードバックの質は変わらない（COM エラーのまま）

### アイデア2：既知成功例の徹底収集（推奨）
**内容：** 過去に動作した既存スクリプトから「同じパターン」の実例を見つけ、
完全にそのパターンをコピーして LLM に渡す
**根拠：** LLMが「理解してから生成」するのではなく「実例を模倣」する形にする
**効果が高い理由：** PLS法線ベクトルなど正確な値が含まれている

### アイデア3：知識注入（根本解決・最優先）
**内容：** 不足している具体的な知識をドキュメント化して RAG に追加
- 各名前付きPlate/Profileに対応するPLS法線ベクトル一覧
- BracketType別の必須パラメータ対応表
- FlangeType と BracketType の有効な組み合わせ表

**根拠：** フィードバックの質を改善するより、最初から正しい情報を与える方が効率的
**これが根本解決**

### アイデア4：タスク分割（Geometry Phase + Code Phase）
**内容：**
1. Phase A: EvoShip クエリAPI で実際のジオメトリ情報（法線ベクトルなど）を取得するスクリプト
2. Phase B: Phase Aの出力を使って CreateBracket スクリプトを生成

**根拠：** LLMが「知れない情報」をCOMクエリで取得してからスクリプト生成に使う
**技術的実現性：** EvoShipのAPIにジオメトリクエリが存在するか確認が必要

### アイデア5：エラー診断レイヤーの追加
**内容：** CreateBracketが失敗した時に、失敗前に各パラメータを個別に検証する
事前チェックスクリプトを用意し、どのパラメータが不正かを特定
**根拠：** フィードバック品質を改善することで、LLMが方向性を持って修正できる
**限界：** EvoShip APIにパラメータ検証機能があるかどうかに依存

---

## 結論

| 問い | 答え |
|-----|------|
| 4回以上繰り返せば完成するか？ | 低確率。収束の保証なし |
| LLMの知識が根本的に不足しているか？ | Yes。特にジオメトリ依存情報と未文書化の必須パラメータ |
| 最も効果的な改善策 | アイデア2（実例完全模倣）+ アイデア3（知識注入） |
| 次のステップとして実現可能なもの | アイデア1（個別テスト）でまず失敗パターンを特定する |

**本質的な問題：**
「LLMが答えを知らないのに、答えのない問いを繰り返させている」
解決策は「答えを教えること」であり、「繰り返し回数を増やすこと」ではない。
