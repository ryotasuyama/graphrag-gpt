#### 修正・除外した候補
- **除外: 前回 #30**
  - `[profile62[0]+",FL"]` ↔ `[profile28[0]+",FL"]`
  - `HK.Casing.Wall.Fore.DL01.CDP` ↔ `HK.Casing.Deck.C.DL01.FP`

#### 除外理由
- 実行時に `CreateBracket` で COM 例外が発生しており、**この組合せは不成立**と判断するのが妥当
- Fore/Aft 方向ルール上は  
  - `Wall.Fore.DL01.*` → `Deck.*.DL01.F / FP`  
  に一致しており、**名前規則だけでは許可対象**
- しかし、`1501` の FL-FL 直結としては、**幾何拘束または面の取り方が Deck.C.DL01.FP 側で安定していない**可能性が高い
- 同じ `Wall.Fore.DL01.CDP` でも
  - `profile61` = `Deck.D.DL01.FP`
  - `profile71` = `Deck.B.DL01.FP`
  - `profile109` = `Deck.A.DL01.FP`
  との組合せは成立候補であり、**Deck.C のみ失敗**している
- よって、原因は Fore/Aft 不整合ではなく、**Deck.C の当該 DL01.FP と Wall.Fore.DL01.CDP の交差条件・面ペア条件が 1501 の前提を満たしていない**点にあると判断する

#### 代替提案
- 代替は**新規の未検証候補を増やさず**、すでに成功済み・成立実績ありの候補を維持する
- 特に `DL01 Fore` 系は以下で十分に網羅される
  - `#1` Wall.Fore.DL01 ↔ Deck.D.DL01.FP
  - `#28` Wall.Fore.DL01 ↔ Deck.A.DL01.FP
  - `#29` Wall.Fore.DL01 ↔ Deck.B.DL01.FP
- これにより、失敗候補 #30 を除外しても **41候補** を維持でき、**30個以上**の要件を十分満たす

### ループ構造方針

前回方針を維持しつつ、**今回失敗した Wall.Fore.DL01.CDP ↔ Deck.C.DL01.FP のような、名前規則上は適合しても実幾何で不成立となる組を除外**します。

#### ループ1: Wall Fore/Aft の DL 系ブラケット
- Wall を `Fore` と `Aft` に分離
- DL番号単位で候補化
- Deck 候補は Deck A/B/C/D の DL プロファイル
- 必須チェック
  - `Wall.Fore.DLxx.*` → `Deck.*.DLxx.F / FP`
  - `Wall.Aft.DLxx.*` → `Deck.*.DLxx.A / AP`
- さらに今回の再分析結果として、
  - **方向整合だけで採用せず**
  - **同型成立実績または個別検証で通った組のみ採用**
- `Wall.Fore.DL01.CDP ↔ Deck.C.DL01.FP` はこの例外として除外

#### ループ2: Side Wall ↔ Deck 板ブラケット
- Side.CDP 系で Deck.D に届くものを対象
- 形式固定
  - `BracketType=1505`
  - `Surfaces1=["PLS",...,solid2]`
  - `Surfaces2=[side_profile+",FL"]`
  - `Sf1EndElements=["PLS",... ,profile17[0]]`
- 参考スクリプトの成立実績が最も多く、安全性が高い

#### ループ3: Side Wall ↔ Deck FR プロファイルブラケット
- 直接 1501 接続は**参考スクリプトに成立実績のある組のみ**
- 採用可
  - `profile65 ↔ profile64`
  - `profile43 ↔ profile42`
  - `profile57 ↔ profile90`
  - `profile95 ↔ profile96`
  - `profile106 ↔ profile112`
  - `profile101 ↔ profile110`
  - `profile43 ↔ profile78`
- 採用不可
  - FR一致のみを根拠にした未実績接続
  - 実行時に COM 例外が出た未安定組
  - Wall/Deck が名前上は対応していても、実交差が不安定な FL-FL 直結

### パラメータ依存関係

#### 1. BracketType の使い分け
- `1501`
  - Wall.DL ↔ Deck.DL
  - Side.FR ↔ Deck.FR
  - ただし Side↔Deck.FR は**成立実績のある組だけ**
- `1505`
  - Side.FR ↔ Deck 板面
  - `BracketParams=["200"]`
  - `Sf1EndElements` 必須

#### 2. Thickness
- `1002` 同士の DL 接続: `8`
- `1201` を含む DL02 系: `12`
- Side↔Deck板 `1505`: `10`
- `1003` を含む FR 接続: `8～9`
- 今回失敗した `profile62 ↔ profile28` は厚み値自体より、**接続位置・面ペアの不整合**が主因と判断

#### 3. Surface型
- 通常 profile 接続
  - `[profileX[0]+",FL"]`
- Deck FR 支持ブラケット
  - `[profile18[0]+",WF"]`
  - `Sf1EndElements=[profile18[1]+",FR"]`
- 板面基準
  - `["PLS","False","False","0","-0","-1",solid2]`
- `1201` 系では `[0] / [1]` の使い分け必須

#### 4. DimensionType
- 一般 profile 接続
  - `1531 / ["200","15"]`
- FR大物
  - `1531 / ["250","15"]` または `["300","15"]`
- 板面基準 1505
  - `1541 / ["0","100"]`
  - `Sf1EndElements` 必須
- DL02 の大型接続
  - `1531 / ["800","15"]`

#### 5. 今回の失敗要因の整理
- `1501` の FL-FL 接続であっても、**Fore/Aft 方向整合だけでは不十分**
- `Wall.Fore.DL01.CDP ↔ Deck.C.DL01.FP` は命名規則上は妥当だが、実際には `CreateBracket` が失敗している
- したがって Wall Fore/Aft 系でも
  - **方向整合**
  - **Deck 側実在**
  - **成立実績**
  - **実幾何での安定性**
  の4点を満たす組のみ採用する
- 特に同一DLでも、Deck A/B/D は成立して Deck C だけ失敗するケースがあるため、**甲板帯ごとの個別成立性確認が必要**

### 注意事項

- **Fore/Aft 整合ルールは必須**
  - `Wall.Fore.DLxx.*` → `Deck.*.DLxx.F / FP` のみ
  - `Wall.Aft.DLxx.*` → `Deck.*.DLxx.A / AP` のみ
- 片側方向の Deck プロファイルしかない場合、反対側 Wall 候補は除外する

- **Wall 側と Deck 側の Fore/Aft は候補化時に毎回検証する**
  - 今回の失敗候補は方向規則には適合していた
  - そのため、**方向整合は必要条件だが十分条件ではない**ことを再確認する

- **Side Wall は方向制約なし**
  - ただし Deck 側相手の存在確認は必須
  - さらに、**成立実績のある接続型かどうか**を必ず確認すること
